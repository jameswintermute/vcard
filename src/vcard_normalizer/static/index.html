STEP 1 — UID CLEANUP — replace vendor-issued UIDs with clean vCard Studio ones
Proton, Apple, and Google generate their own proprietary UIDs (e.g. proton-web-9726a5f6…, AB3F2C…). This replaces them all with clean vcard-studio-<uuid> identifiers and assigns UIDs to any card that has none. The change is saved to the checkpoint immediately.
STEP 2 — PHONE FORMATTING — reformat all numbers to international spacing
Converts stored numbers to +44 1642 714500 format (GOV.UK style, no brackets). Run after import if your source had inconsistent formatting.
STEP 3 — COUNTRY NAMES — normalise to standard country names
Merges variants like UK, England, Britain → United Kingdom. Full editor is also available in Insights.
STEP 4 — AUTO-GENDER FROM PREFIX — infer gender from name prefix
Sets GENDER:M for contacts with prefix Mr or Master, and GENDER:F for Mrs, Ms, or Miss. Only updates contacts where gender is not already set. Skips organisations.
STEP 4b — AUTO-PREFIX FROM GENDER — infer missing prefix from gender
Sets Mr for male individuals with no prefix, and Ms for female individuals with no prefix. Only updates contacts where prefix is not already set. Skips organisations.
STEP 4c — APPLE NAME FIX — add first/last name where only a display name exists
Apple / iOS ignores the display name (FN) and builds the contact label from the structured N field (first + last). Contacts that only have a display name will appear as just their prefix — e.g. Ms — in the iOS Contacts app. This step sets a X-IOS-GIVEN / X-IOS-FAMILY override that is injected into the Apple export only — the real card data (KIND, name fields) is never modified.
STEP 5 — AUTO-REVIEW — scan for contacts that may need attention
Checks all loaded contacts for common data quality issues
⚠ VENDOR CLEANUP — remove all proprietary traces
Strips all vendor-specific X-* properties and PRODID from every card, including any traces left by Apple, Google, Proton, or vCard Studio itself. This makes the output truly vendor-neutral. Optionally also resets any "not required" field markers you have set.

· Rule-based auto-tagging from local/vcard.conf patterns
· Proprietary Apple/Google fields stripped
· Vendor UIDs (Proton, Apple, Google) replaced on import with vcard-studio-<uuid>

1 Combined file · vCard 4.0 recommended
All contacts in a single .vcf file using the full vCard 4.0 standard. Preserves all fields including linked relationships, gender, kind, and custom fields. Best suited for Linux, ProtonMail, Thunderbird, and any modern CardDAV-based application.
2 Combined file · Apple / iOS compatibility mode iCloud · iPhone
All contacts in a single .vcf file using vCard 3.0 — the format iCloud and iOS actually accept reliably. Fields that are 4.0-only (linked relationships, gender, kind, anniversary) are not lost — they are preserved in each contact's Notes field as a compact ---vCard Studio--- block, ready for re-import later. Phone numbers are written as plain +44... values rather than URI format, which resolves silent import failures on iPhone.
3 Individual files · vCard 4.0 sharing
Each contact exported as its own .vcf file in cards-out/vcard-studio-<date>/, named vcard-<date>-<LastName>-<FirstName>.vcf. Full vCard 4.0 format. Ideal for sharing individual contacts or importing into applications that expect one file per contact.
4 Individual files · Apple / iOS compatibility mode iCloud · iPhone
Each contact as its own vCard 3.0 file in cards-out/apple-<date>/. Applies the same Apple compatibility rules as Option 2 — plain phone numbers, 4.0-only fields preserved in Notes. Use this when you need to share or import individual contacts into the Apple ecosystem.
